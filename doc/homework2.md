# FlashMall 实验简要说明

## 1. 实验目标

本实验基于 FlashMall 微服务项目，完成以下内容：

- 使用 Docker Compose 对数据库、Redis、Nacos、后端服务和 Nginx 进行容器化部署
- 启动多个商品服务实例，通过 Nginx 实现负载均衡
- 编写简单前端页面，并在 Nginx 中实现动静分离
- 引入 Redis 缓存优化商品详情接口
- 使用 JMeter 对静态资源和后端接口进行压力测试

## 2. 系统结构

系统主要由以下部分组成：

- MySQL：存储用户、商品、库存、订单等业务数据
- Redis：缓存商品详情数据
- Nacos：服务注册与发现
- flashmall-user、flashmall-order、flashmall-stock：基础业务服务
- flashmall-goods：商品服务，启动两个实例，端口分别为 `8081`、`8082`
- Nginx：对外统一提供 `80` 端口，负责静态资源访问和后端代理转发

## 3. 负载均衡设计

Nginx 对商品服务提供了多种负载均衡方式：

- 轮询：`/api/goods/**`
- 最少连接：`/api/lb/least/goods/**`
- IP Hash：`/api/lb/iphash/goods/**`
- 一致性哈希：`/api/lb/urihash/goods/**`

其中轮询和最少连接适合观察请求是否被较均匀地分配到两个商品服务实例，`ip_hash` 和一致性哈希适合观察固定请求更稳定命中某个实例的效果。

## 4. 动静分离设计

前端静态页面 `index.html`、CSS 和 JS 由 Nginx 直接提供访问，不经过后端服务。  
商品列表和商品详情接口则通过 Nginx 反向代理到后端商品服务或网关服务，从而实现动静分离。

这种方式的优点是：

- 静态资源访问速度快
- 减少后端服务压力
- 便于前后端分离部署

## 5. 缓存方案

本实验在商品详情接口中引入 Redis 缓存，采用 Cache Aside 模式：

1. 先查询 Redis
2. 缓存未命中再查数据库
3. 将数据库结果写回 Redis

针对常见缓存问题，采用了以下方案：

- 缓存穿透：对不存在的商品写入空值缓存，避免无效请求持续访问数据库
- 缓存击穿：对热点 Key 使用 Redisson 分布式锁，并在加锁后再次检查缓存，防止高并发同时回源数据库
- 缓存雪崩：给缓存过期时间增加随机值，避免大量缓存同一时间失效

## 6. 压测结果

### 6.1 静态页面压测

使用 JMeter 对首页静态文件进行压测，结果如下：

- 总请求数：2400
- 平均响应时间：4 ms
- 吞吐量：467 次/秒
- 错误率：0%

### 6.2 商品详情接口压测

对商品详情接口 `/api/goods/detail/1` 进行压测，结果如下：

- 总请求数：1000
- 平均响应时间：3 ms
- 吞吐量：201.7 次/秒
- 错误率：0%

## 7. 结果分析

从实验结果可以看出：

- 静态资源由 Nginx 直接返回，吞吐量更高，说明动静分离效果明显
- 商品详情接口平均响应时间较低，说明 Redis 缓存已经生效，减少了数据库压力
- 轮询和最少连接模式下，两个商品服务实例的处理请求数应大致接近，可通过查看后端日志中的 `handledRequestCount` 进行验证
- 在 `ip_hash` 模式下，由于请求来源 IP 相同，请求更容易集中命中同一个实例

## 8. 实验结论

本实验完成了微服务项目的容器化部署，并验证了 Nginx 负载均衡、动静分离和 Redis 缓存优化的实际效果。  
实验结果表明，静态资源访问性能明显优于动态接口，而动态接口在接入缓存后也具有较好的响应速度和稳定性。整体系统能够较好地满足课程实验对容器环境、负载均衡、缓存和性能测试的要求。
